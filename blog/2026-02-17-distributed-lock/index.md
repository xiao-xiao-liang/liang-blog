在单机应用时代，我们使用 `synchronized` 或 `ReentrantLock` 就可以轻松解决多线程竞争共享资源的问题。但随着微服务架构的普及，应用被部署在多台服务器上，本地锁就失效了。这时候，就需要一把“分布式锁”来保证跨进程、跨主机的互斥访问。

本文将带你深入了解分布式锁的常见实现方案，特别是 Redis 方案的演进，以及工业级组件 Redisson 的底层原理。

## 1. 分布式锁的核心要求

一个靠谱的分布式锁，通常需要满足以下条件：

1.  **互斥性**：在任意时刻，只能有一个客户端持有锁。
2.  **防死锁**：即使持有锁的客户端崩溃或网络断开，锁也能自动释放（通常通过过期时间实现）。
3.  **可重入性**：同一个客户端可以多次获取同一把锁。
4.  **高性能/高可用**：加锁和解锁需要开销小，且服务要稳定。

## 2. 常见的实现方案

### 2.1 基于数据库实现

最简单的想法是利用数据库的**唯一索引（Unique Constraint）**。

*   **加锁**：插入一条记录（比如 `INSERT INTO locks(name) VALUES('my_lock')`）。如果插入成功，则获取锁；如果报唯一键冲突，则获取失败。
*   **解锁**：删除该记录。

**优点**：
*   简单，不需要引入中间件。

**缺点 & 坑**：
*   **性能差**：数据库操作涉及到磁盘 IO，并发能力弱。
*   **无自动失效**：如果持有锁的节点挂了，记录没删掉，就死锁了。（需要搞个定时任务清理）。
*   **不可重入**：实现复杂。

### 2.2 基于 Redis 的基础实现 (`SETNX`)

Redis 是内存数据库，性能极高，是实现分布式锁的首选。

#### 版本 1.0：SETNX + EXPIRE

早期的 Redis 分布式锁通常是这样写的（伪代码）：

```java
if (redis.setnx("lock_key", "value") == 1) {
    redis.expire("lock_key", 30); //设置30秒过期
    try {
        // 业务逻辑
    } finally {
        redis.del("lock_key");
    }
}
```

**坑在哪里？**
`setnx` 和 `expire` 是两条命令，如果 `setnx` 成功后服务器挂了，`expire` 没执行，这把锁就**永不过期**了（死锁）。

#### 版本 2.0：原子命令

从 Redis 2.6.12 开始，`SET` 命令支持参数组合，实现了原子性：

```bash
SET lock_key unique_value NX PX 30000
```
*   `NX`：Not eXists，只有 key 不存在时才设置。
*   `PX 30000`：30秒后自动过期。

这样就解决了原子性加锁的问题。

#### 还可以更好吗？存在的隐患

虽然 2.0 版本解决了死锁，但还有两个大问题：

1.  **误删锁**：A 线程拿了锁，设置 30s 过期。由于业务执行慢，用了 40s。第 30s 时锁自动过期，B 线程拿到了锁。第 40s 时 A 线程执行完，把 `lock_key` 删了——**A 删了 B 的锁**。
    *   **解决**：Value 值设置为 UUID，删除前判断 Value 是否就是自己的 UUID。注意：判断和删除也必须是原子操作（用 Lua 脚本）。

2.  **锁过期时间不好设**：
    *   设短了：业务没跑完锁就丢了，不安全。
    *   设长了：万一节点挂了，其他节点要等很久。
    *   **痛点**：我们很难预估业务到底要跑多久。

## 3. Redisson：工业级解决方案

Redisson 是一个在 Redis 基础上实现的 Java 驻内存数据网格（In-Memory Data Grid）。它提供的 `RLock` 完美解决了上述所有痛点。

### 3.1 简单易用

```java
RLock lock = redisson.getLock("myLock");
try {
    // 1. 加锁，默认过期时间 30s
    lock.lock();
    // 业务逻辑
} finally {
    lock.unlock();
}
```

### 3.2 核心原理揭秘

Redisson 的强大在于它通过 **Lua 脚本** 保证了复杂操作的原子性，并引入了 **Watchdog（看门狗）** 机制。

#### (1) 加锁机制（Lua 脚本）

Redisson 在加锁时，会执行一段 Lua 脚本。大体逻辑是：
1.  判断 key 是否存在？
2.  不存在 -> `hset` 设置 hash 结构（Key=锁名, Field=UUID+线程ID, Value=重入次数 1），并设置过期时间。
3.  存在 -> 判断 Field 是否是当前线程？
    *   是 -> 重入次数 +1，重置过期时间。
    *   否 -> 返回当前锁的剩余过期时间（加锁失败）。

**注意**：Redisson 使用 Hash 结构来存储锁，这是为了支持**可重入**。

#### (2) Watchdog 看门狗（自动续期）

这是 Redisson 最核心的设计。

*   **场景**：可以不指定 LeaseTime（过期时间），默认是 30 秒（`lockWatchdogTimeout`）。
*   **机制**：加锁成功后，Redisson 会在后台启动一个定时任务（TimeTask），每隔 `lockWatchdogTimeout / 3` （默认 10秒）检查一下，如果当前线程还持有这把锁，就重新把过期时间设为 30 秒。
*   **好处**：
    1.  业务没跑完，锁永远不会过期。
    2.  业务跑完了，手动 unlock，看门狗任务取消。
    3.  机器宕机了，看门狗也没了，30秒后锁自动释放。

### 3.3 Redisson 的“坑”与注意事项

虽然 Redisson 很好用，但也有极端情况：

*   **主从切换导致锁丢失**：
    Redis 主从复制是异步的。如果 Client A 在 Master 拿到了锁，还没同步到 Slave，Master 挂了。Slave 升级为 New Master，Client B 也能拿到如同名锁。
    *   **解决**：使用 **RedLock** 算法（向多个独立的 Redis 实例请求锁，过半成功才算成功）。但在实际生产中，RedLock 运维成本高，且仍有争议（争议主要来自 Martin Kleppmann 与 Antirez 的论战）。通常大家还是接受单点故障带来的极小概率锁丢失，或者改用 Zookeeper。

## 4. Zookeeper 分布式锁

Zookeeper (ZK) 是 CP 模型（强一致性），天然适合做分布式协调。

*   **原理**：利用 ZK 的**临时顺序节点**。
    1.  客户端在 `/locks` 下创建一个临时顺序节点。
    2.  判断自己是不是序号最小的节点？
        *   是 -> 获得锁。
        *   否 -> 监听（Watch）前一个节点。
    3.  前一个节点删除（锁释放或对方挂了），触发 Watch 事件，自己再次判断是否是最小。

**对比 Redis vs Zookeeper**：

| 维度 | Redis (Redisson) | Zookeeper |
| :--- | :--- | :--- |
| **一致性** | 弱（AP模型，主从切换可能丢锁） | 强（CP模型，保证一致性） |
| **性能** | 极高（内存操作） | 中等（需要写磁盘日志，Leader同步） |
| **死锁风险** | 靠过期时间兜底 | 靠临时节点（Session断开自动删除） |
| **适用场景** | 高并发业务（秒杀、库存扣减） | 对一致性要求极高的场景 |

---

## 5. 总结

在大多数互联网业务场景下，**Redis (Redisson)** 是性价比最高的选择。

*   如果是**相亲相爱**的内部系统，追求极致并发，允许极端情况下（主从切换）有极小概率的并发冲突 -> **选 Redis**。
*   如果是**金融级**场景，或者涉及到钱，绝不允许任何并发差错 -> **选 Zookeeper** 或 **数据库悲观锁**（如果并发量不大的话）。

**最后建议**：不要自己造轮子去写 Redis 分布式锁（SETNX），坑太多了。直接用 Redisson，省心省力！
